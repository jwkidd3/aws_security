Security IN the Cloud — Part 3

DDoS mitigation · Incident response essentials

🌊 DDoS — What You're Up Against

A DDoS (Distributed Denial of Service) attack saturates your capacity so legitimate users can't get through.

  • L3/L4 volumetric — packet floods saturating bandwidth or connection tables
  • L7 application — valid-looking HTTP requests that exhaust app compute, DB connections, or third-party quotas
  • Reflection / amplification — DNS, NTP, memcached misused to amplify packet size
  • The goal is not usually to breach data — it's to make you unreachable

🛡️ AWS DDoS Mitigation — The Layers

  • AWS backbone & edge — massive capacity absorbs volumetric traffic before it reaches you
  • Shield Standard — always on, free, automatic L3/L4 protection
  • Shield Advanced — paid tier: cost protection, 24×7 DDoS Response Team, L7 detection, health-based detection
  • CloudFront & Global Accelerator — push ingress to the edge, nearest to users
  • AWS WAF — rate-based rules and managed rule groups for L7 floods
  • Route 53 — Anycast DNS with built-in DDoS resilience

🏗️ Architecting for DDoS Resilience

  • Put CloudFront in front of every public-facing site — terminate TLS at the edge
  • Use auto-scaling + load balancers so capacity can absorb a surge
  • Protect origins — restrict origin access so the ALB/ASG only accepts traffic from CloudFront
  • Enforce rate limits on login, search, and any expensive endpoint
  • Keep a runbook — know who to call, what to turn on, what to disable

🚨 Incident Response — Cloud-Native Reality

  • The cloud changes IR: every control is an API, every action leaves a log, every resource can be rebuilt
  • IR is a capability, not a document — it must be practiced
  • Plan for both infrastructure events (compromised EC2, leaked key) and service events (S3 misconfiguration)
  • The NIST framework applies: Prepare → Detect & Analyze → Contain → Eradicate → Recover → Lessons Learned

🧰 Prepare — Before Anything Breaks

  • A dedicated security account for logs, findings, and forensic tools — isolated from production
  • Break-glass IAM roles — pre-provisioned, heavily-audited, high-privilege roles responders can assume under emergency
  • Contact & escalation lists — on-call, legal, PR, AWS support, executive sign-off
  • Runbooks (SSM documents) for common tasks: isolate an instance, rotate a key, disable a user
  • Game days — simulate real incidents; rotate the responder

🔍 Detect & Analyze

  • GuardDuty, Security Hub, and Config findings are the usual triggers
  • Amazon Detective — automatically builds investigation graphs across CloudTrail and VPC flows
  • CloudTrail Lake / Logs Insights — ad-hoc queries across months of audit data
  • Answer these fast: what happened, when, which principal, from what IP, which resources touched?

🧯 Contain & Eradicate

Common cloud-native containment actions:

  • Revoke credentials — disable an IAM user, remove access keys, force MFA re-auth
  • Rotate — KMS keys, secrets, DB passwords, API tokens
  • Quarantine an instance — swap its security group to one that denies all inbound/outbound, take an EBS snapshot first
  • Block an IP — WAF IP-set or NACL rule
  • Invalidate sessions — Cognito global sign-out, revoke session tokens
  • Kill the stack — in some cases, terminating and redeploying from known-good IaC is fastest

📈 Recover & Learn

  • Recover — redeploy from clean IaC, restore from tested backups, validate data integrity, monitor closely
  • Post-incident review — blameless, within 5 business days
  • Feed findings back into: detections, runbooks, SCPs, training, architecture
  • Track time-to-detect and time-to-contain — these are the numbers that matter

🧪 Forensic Hygiene

  • Preserve before you touch — snapshot EBS volumes, export logs, tag the evidence
  • Store evidence in a separate, read-only account with Object Lock
  • Don't power off a live instance before capturing memory if relevant
  • Legal & compliance usually need to be in the loop before anything leaves AWS
