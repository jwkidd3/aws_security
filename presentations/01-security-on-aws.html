What Makes Cloud Security Different

  • The perimeter is an API, not a data-center door — every control is an API call with an audit log
  • Everything is identity-driven — IAM is the new firewall
  • Infrastructure is code, so policy is code too — guardrails scale
  • AWS owns and secures the physical layer; you secure what you put on top
  • You can rebuild a compromised environment in minutes — if it's automated

AWS Security Design Principles

From the Well-Architected Security Pillar.

  • Implement a strong identity foundation — least privilege, central identity, no long-lived keys
  • Enable traceability — log everything, alert in near-real time
  • Apply security at all layers — edge, VPC, host, app, data
  • Automate security best practices — guardrails that scale
  • Protect data in transit and at rest — encrypt, classify, tokenize
  • Keep people away from data — reduce direct access and manual handling
  • Prepare for security events — IR runbooks, game days

The Shared Responsibility Model

A foundational AWS concept: the security line depends on the service.

AWS — security OF the cloud
  • Regions, AZs, edge locations
  • Data-center physical security
  • Host hardware, hypervisor, storage
  • Network fabric
  • Managed-service internals
You — security IN the cloud
  • Your data & classification
  • IAM users, roles, and policies
  • OS, apps, patches (where applicable)
  • Network config (VPCs, SGs)
  • Encryption choices & key use

The Line Shifts by Service

  • EC2 (IaaS) — AWS secures host & hypervisor; you patch the OS, harden the image, own network rules
  • RDS / ECS / EKS (managed) — AWS handles the engine; you own schema, IAM, network, encryption keys
  • S3 / DynamoDB / Lambda (abstracted) — AWS runs the service entirely; you own access policies, data, encryption choice
  • When something breaks, the first question is always: which side of the line does this sit on?

The AWS Shared Controls Model

Some controls are shared — both sides play a part.

  • Patch management — AWS patches infrastructure; you patch guest OS & app
  • Configuration management — AWS configures infra; you configure guest OS, DB, app
  • Awareness & training — AWS trains its staff; you train yours
  • A "shared" control doesn't mean split 50/50 — read the service documentation for specifics

Common AWS Security Services You'll Meet Today

  • IAM (Identity and Access Management) — who can do what on which resources
  • KMS (Key Management Service) — create and control encryption keys
  • CloudTrail — records every API call made in your account
  • CloudWatch — metrics, logs, and alarms
  • Config — tracks resource configuration and compliance over time
  • GuardDuty — continuous threat-detection across accounts
  • Security Hub — central dashboard that aggregates findings
  • WAF & Shield — L7 web attacks and DDoS mitigation
  • Secrets Manager — store, rotate, and retrieve secrets
