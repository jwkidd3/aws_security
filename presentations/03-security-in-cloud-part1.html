Security IN the Cloud — Part 1

Identity & Access Management · Data Protection Essentials

Authentication vs. Authorization

Two different questions — often conflated.

  • Authentication = who are you? — proving identity (password, MFA, federation)
  • Authorization = what can you do? — what the identity is allowed to do once in
  • A user can be authenticated (logged in) and still denied (not authorized)
  • IAM handles both — but most policy work is authorization

IAM — The Building Blocks

  • User — a long-lived identity for a human or legacy app (avoid where possible)
  • Group — a container of users; attach policies to groups, not users
  • Role — an identity that workloads or federated humans assume temporarily (short-lived creds)
  • Policy — a JSON document granting or denying actions on resources
  • Principal — who/what is making the request (user, role, service)
  • Permission boundary — a maximum-allowed policy that caps what a role can ever receive
  • SCP — org-level guardrail: even root in a member account can't exceed it

Identity Federation & IAM Identity Center

  • IAM Identity Center (formerly AWS SSO) — central sign-on across all AWS accounts
  • Federate from your corporate IdP (Okta, Entra ID, Ping, Google Workspace) with SAML or OIDC
  • Humans log in once → assume short-lived roles in each account — no per-account IAM users
  • Machine workloads use IAM roles (EC2 instance profiles, ECS task roles, Lambda execution roles)
  • Goal: no long-lived access keys anywhere you can avoid them

Root Account & MFA

  • The root user is the original all-powerful account — lock it away
  • MFA required on root — prefer a hardware key (FIDO2)
  • Remove all root access keys; use root only for billing and rare recovery tasks
  • Every human account must have MFA — no exceptions
  • Do daily work through IAM Identity Center or IAM roles, never as root

Anatomy of an IAM Policy

{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::reports-prod/*",
  "Condition": {
    "StringEquals": { "aws:RequestedRegion": "us-east-1" },
    "Bool":         { "aws:MultiFactorAuthPresent": "true" }
  }
}
  • Effect — Allow or Deny (explicit Deny always wins)
  • Action — API call(s), e.g., s3:GetObject
  • Resource — ARN(s) the actions apply to
  • Condition — constraints: region, source IP, MFA, tag values…

Least Privilege & Policy Tooling

  • Least privilege — grant only the actions actually used, on the specific resources needed
  • Avoid "Action": "*" and "Resource": "*" in anything that runs in production
  • IAM Access Analyzer — flags policies that grant access outside your account or org; can generate a policy from observed CloudTrail activity
  • IAM Policy Simulator — test a proposed policy against real API calls before deploying
  • Last Accessed Info in the IAM console — see which services a role actually used (and prune)

Resource-Based Policies

Policies attached to the resource rather than the principal — cross-account access without assumed roles.

  • S3 bucket policies — who can read/write objects in this bucket
  • KMS key policies — who can use this key (required, not optional)
  • SQS / SNS / Lambda resource policies — allow other accounts or services to invoke
  • A request is allowed if either side allows it — unless there is an explicit Deny

Data Protection — The Big Picture

  • Classify first — Public, Internal, Confidential, Restricted — controls follow the class
  • Encrypt at rest — most AWS services encrypt by default; you choose which key
  • Encrypt in transit — TLS everywhere, including between services inside your VPC
  • Limit direct access — pull data through APIs, not through human console sessions
  • Protect backups — encrypted, versioned, tested, and often in a separate account

KMS — Key Management Service

  • KMS key (formerly CMK) — the logical key you grant usage of via key policies
  • AWS-managed key — AWS creates and controls; simple, no extra cost
  • Customer-managed key (CMK) — you own the key policy, rotation, grants, audit
  • Envelope encryption — KMS encrypts a per-object data key; the data key encrypts the data. Efficient at scale.
  • Every use of a KMS key is logged in CloudTrail
  • CloudHSM — dedicated FIPS 140-2 Level 3 HSMs when you must hold the keys

Encryption in Transit

  • TLS on every endpoint — all AWS public endpoints already require HTTPS
  • ACM (AWS Certificate Manager) — free public certs, automatic renewal for ELB, CloudFront, API Gateway
  • ACM Private CA — private certificates for service-to-service mTLS
  • Pin modern protocols (TLS 1.2+), disable legacy ciphers at the load balancer
  • Database traffic too — enforce SSL on RDS, TLS on Redis, mTLS on MSK

Secrets Management

  • Never commit credentials to code, env files, or AMIs
  • Secrets Manager — store secrets, rotate them automatically, retrieve via SDK at runtime
  • SSM Parameter Store — lighter-weight, cheaper; SecureString parameters are KMS-encrypted
  • Workloads fetch secrets using their IAM role — not a bootstrap password
  • Rotate on a schedule — and after any suspected exposure

Data Discovery & Classification

  • Amazon Macie — ML-based discovery of sensitive data (PII, PHI, secrets) in S3
  • Flags open buckets, unencrypted buckets, public-access misconfigurations
  • S3 Object Lock + versioning — WORM storage for compliance data
  • S3 Block Public Access — account-level switch; turn it on everywhere unless you have a reason not to
