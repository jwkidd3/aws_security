Data Protection & KMS

Module 3 — Security IN the Cloud · Part 1

Objective

Choose the right encryption-at-rest option for each component in a fictional workload, decide when AWS-managed keys are sufficient vs. customer-managed keys, and review a KMS key policy for over-broad grants.

Terms You'll See

  • KMS — AWS Key Management Service
  • KMS key — logical key you grant use of via a key policy
  • AWS-managed key — AWS creates and rotates; simple, free
  • Customer-managed key (CMK) — you own the key policy, rotation, grants, and audit trail
  • Envelope encryption — KMS encrypts a per-object data key; data key encrypts the data
  • Key policy — resource policy attached to a KMS key (required, not optional)
  • Secrets Manager — AWS service that stores and rotates secrets
  • Object Lock — S3 WORM (write-once, read-many) for compliance storage

Scenario — "HealthMetric" Workload

A regulated healthcare analytics workload running in one AWS account.

  • S3 bucket holding raw patient data exports (PHI)
  • RDS PostgreSQL holding reporting data (derived, pseudonymized)
  • EBS volumes attached to EC2 processing nodes
  • Secrets Manager storing DB connection strings & third-party API keys
  • CloudTrail logs delivered to a separate log-archive bucket

Task 3 — Review This Key Policy

{
  "Version": "2012-10-17",
  "Id": "phi-bucket-cmk",
  "Statement": [
    {
      "Sid": "EnableRoot",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowEveryone",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "kms:Encrypt","kms:Decrypt","kms:GenerateDataKey",
        "kms:DescribeKey","kms:ReEncrypt*"
      ],
      "Resource": "*"
    }
  ]
}

List at least 3 problems. Which statement is dangerous, and what would you change?

Task 3 — Answer Key

  • "Principal": "*" allows any AWS principal, including other accounts — this is the headline bug
  • No kms:ViaService condition → the key can be used from any service, not just S3
  • No aws:SourceAccount / kms:CallerAccount condition → no account-scope guardrail
  • Including kms:ReEncrypt* with wildcards is often broader than needed
  • Fix: restrict Principal to specific role ARNs, add "kms:ViaService": "s3.us-east-1.amazonaws.com", and scope actions to what the workload actually needs

Console Exploration (read-only)

Navigate to these in the AWS Console. Do not create, edit, or save anything.

  1. KMS → AWS managed keys — scan names like aws/s3, aws/rds; note they're per-service
  2. Open one AWS-managed key → view the Key policy tab; notice the default grants
  3. S3 → any bucket → Properties → Default encryption; note SSE-S3 vs. SSE-KMS
  4. Secrets Manager → Secrets; open one to view rotation configuration (do not start a rotation)
  5. Macie → Summary — browse the kinds of findings it produces on S3 buckets
