Lab 4

Network Security Design

Module 4 — Security IN the Cloud · Part 2

🎯 Objective

Design a VPC and security-group graph for a 3-tier web app. Then review a handed-to-you configuration and find the mistakes.

📚 Terms You'll See

  • VPC — your isolated virtual network in AWS
  • Subnet — slice of the VPC's IP range tied to one AZ
  • IGW (Internet Gateway) — route target that makes a subnet public
  • NAT Gateway — egress-only internet for private subnets
  • Security group — stateful firewall on an ENI; default-deny inbound
  • NACL — stateless subnet firewall; default-allow
  • VPC endpoint — private path from your VPC to AWS services
  • ALB — Application Load Balancer (L7)

🏗️ Scenario — 3-Tier Web App

  • Public ALB terminates HTTPS
  • App tier: auto-scaling EC2 in private subnets, one region, two AZs
  • Data tier: RDS PostgreSQL, Multi-AZ, no public endpoint
  • App needs to reach S3 and Secrets Manager
  • Engineers connect via SSM Session Manager — no SSH keys

🗺️ Task 1 — Subnet Layout

Sketch the VPC. For each subnet, note: name, public/private/isolated, which AZ, and what lives there.

Answer these before moving on:

  1. How many subnets do you need at minimum?
  2. Which subnets have an IGW route; which have a NAT route; which have no internet route?
  3. How do the app instances reach S3 without going through the NAT Gateway?
  4. How do engineers get a shell on an instance with no public IP, no bastion, no SSH?

🔗 Task 2 — Security-Group Graph

Name each SG and list its inbound rules (source → port).

Security GroupInbound Rules
sg-alb
sg-app
sg-db
sg-vpce (for interface endpoints)

Prefer SG-to-SG references over CIDRs. Outbound defaults to allow-all unless you restrict it — consider whether you should.

✅ Task 2 — Sample Answer

  • sg-alb — inbound 443 from 0.0.0.0/0 (and 80 if you redirect to 443)
  • sg-app — inbound 8080 from sg-alb
  • sg-db — inbound 5432 from sg-app
  • sg-vpce — inbound 443 from sg-app

SG-to-SG references make the "app may reach DB" relationship explicit and survive IP changes.

🕵️ Task 3 — Review This Config

A junior engineer proposed this. Find at least five issues.

VPC: 10.0.0.0/16
Subnets:
  subnet-public-a (10.0.0.0/24, AZ-a, IGW route)
    - ALB
    - App EC2
    - RDS PostgreSQL

Security groups:
  sg-alb: inbound 443 from 0.0.0.0/0, inbound 22 from 0.0.0.0/0
  sg-app: inbound 0-65535 from 0.0.0.0/0
  sg-db:  inbound 5432 from 0.0.0.0/0

Other:
  - RDS "Publicly accessible" = Yes
  - App instances have public IPs
  - SSH key "app-key.pem" emailed to each developer
  - CloudTrail disabled "to save money"

⚠️ Task 3 — Issues

  • All tiers collapsed into one public subnet — no network segmentation; DB is internet-reachable
  • Single AZ — not even reliability-safe, let alone security-safe
  • ALB SG allows SSH from the internet — SSH does not belong on an ALB at all
  • App SG opens all 65k ports to 0.0.0.0/0
  • DB SG opens PostgreSQL to the internet
  • RDS is publicly accessible
  • App instances have public IPs — should be private + SSM
  • SSH keys distributed via email — replace with SSM Session Manager
  • CloudTrail disabled — there will be no evidence when something goes wrong

🔍 Console Exploration (read-only)

Navigate to these in the AWS Console. Do not create, edit, or save anything.

  1. VPC → Your VPCs → any VPC → Resource map. Trace subnets, route tables, IGW, NAT
  2. VPC → Endpoints. Note which services have Gateway vs. Interface endpoints
  3. EC2 → Security groups. Sort by name; open one and look at its inbound rules
  4. EC2 → Network ACLs. Note the default allow-all and numbered-rule pattern
  5. WAF & Shield → Web ACLs. Browse any existing ACL; notice the rule groups

💬 Discussion

  • Which of these mistakes would be caught by an AWS Config rule today?
    restricted-ssh · vpc-sg-open-only-to-authorized-ports · rds-instance-public-access-check
  • What's the right way to prevent someone from creating an open security group in the first place?
    SCPs + permission boundaries + IaC reviews + policy-as-code in CI
  • Where do VPC endpoints earn their keep beyond security?
    Cost (no NAT egress fees) · performance (shorter path) · reliability (no internet dependency)
