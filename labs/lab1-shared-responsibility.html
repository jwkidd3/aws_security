Shared Responsibility Matrix

Module 1 — Security on AWS

🎯 Objective

For three realistic workloads, decide which side of the AWS Shared Responsibility line each security control sits on. Identify the controls that are genuinely shared — where both AWS and the customer play a part.

📚 Terms You'll See

  • Shared Responsibility Model — AWS secures the cloud; you secure what's in it
  • IaaS / PaaS / SaaS — layers of abstraction; the line shifts as you climb
  • EC2 — virtual server (IaaS) — you own the OS
  • RDS — managed database — AWS owns the engine, you own schema & access
  • S3 — object storage — AWS runs it, you own access & data choices
  • Lambda — run code without servers; AWS patches the runtime
  • KMS — AWS encryption-key service; you choose managed vs. customer-managed keys

🗝️ Answer Key (Discussion)

Quick-reference — your instructor will walk through the reasoning.

  • AWS — physical DC, hypervisor, network fabric, storage media sanitization, managed-service engine patching (RDS, Lambda runtime, DynamoDB), edge TLS infrastructure
  • Customer — guest OS, app code & config, IAM policies, SG rules, bucket/ key policies, log retention, app users, TLS certs you own, responding to findings
  • Shared — patching (AWS patches infra, you patch guest OS), configuration management, IR (AWS detects infra events, you handle workload events), awareness & training

🔍 Console Exploration (read-only)

Navigate to these in the AWS Console. Do not create, edit, or save anything.

  1. Search for AWS Artifact → Reports tab. Browse (don't download) the available SOC, ISO, PCI reports
  2. Open the AWS Health dashboard (Personal Health Dashboard) — the evidence AWS publishes about its side of the line
  3. Open an EC2 instance's Details tab and note which fields are "yours" (Name, IAM role, SGs) vs. AWS's (Hypervisor, Nitro)
