Module 5 — Security IN the Cloud · Part 3

Objective

Walk two realistic GuardDuty findings through a cloud-native IR workflow: detect, analyze, contain, eradicate, recover, learn. List the exact AWS services and actions you would use — and in what order.

Terms You'll See

  • GuardDuty — AWS threat-detection service
  • finding — a detection, scored Low / Medium / High
  • Detective — service that builds investigation graphs from CloudTrail & VPC flow
  • Break-glass role — pre-provisioned emergency IAM role with elevated access
  • Forensic account — isolated AWS account for evidence & analysis
  • IMDS — EC2 Instance Metadata Service; classic SSRF target
  • SSM Session Manager — agent-based shell; no SSH needed
  • EventBridge — rules that route events to Lambda, SSM, SNS, tickets

Scenario A — Credential Exfiltration

GuardDuty finding:

UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS
  Severity: High
  Resource: EC2 instance i-0a12b34c (prod-api account)
  Detail:  Credentials from the instance role's IMDS were
           observed being used from IP 203.0.113.42 (not AWS)

Someone stole temporary credentials off an EC2 instance and is using them from the internet.

Scenario A — Your Tasks

  1. List the first five actions you take, in order
  2. For each action, name the AWS service or API you would use
  3. What do you preserve before you change anything?
  4. Which root cause would you hunt for? (It's almost always one of three things.)

Scenario A — Sample Playbook

  1. Preserve: snapshot the EBS volume(s) of i-0a12b34c (EC2 CreateSnapshot); export recent CloudTrail & VPC Flow Logs for the instance
  2. Contain: replace the instance's security group with a quarantine SG that denies all inbound and outbound
  3. Contain: rotate the instance-role trust — remove the role from the instance profile, then revoke active sessions with aws iam put-role-policy adding a Deny * with aws:TokenIssueTime before now
  4. Analyze: pivot in Detective from the finding to the role's API activity — what else did 203.0.113.42 touch?
  5. Eradicate: terminate the instance, redeploy from known-good AMI/IaC, force IMDSv2, disable metadata on the new instance if the workload doesn't use it

Root cause is typically: SSRF in the app, an over-permissive instance profile, or IMDSv1 still enabled.

Scenario B — S3 Data Exfiltration

Exfiltration:IAMUser/AnomalousBehavior
  Severity: High
  Principal: iam-user/jdoe (developer, last logged in 3 months ago)
  Detail:  91 GB of S3 objects listed + downloaded from
           bucket "customer-exports-prod" in the last hour
           from new country: off-network

A long-dormant IAM user's access keys are being used to pull a customer export bucket to an unknown location.

Scenario B — Your Tasks

  1. What do you do in the first 60 seconds?
  2. Which IAM actions revoke access immediately vs. eventually?
  3. Which CloudTrail fields tell you which keys, which IPs, which user agent?
  4. Who else in the org needs to know — when, and via what channel?
  5. What permanent change do you make so this entire class of incident is harder next time?

Scenario B — Sample Playbook

  1. Deactivate and then delete both access keys on jdoe (iam:UpdateAccessKey / iam:DeleteAccessKey)
  2. Attach an IAM policy to the user denying "*":"*"; disable console access
  3. Preserve: query CloudTrail Lake for all actions by jdoe in the last 7 days — export results
  4. Review S3 server access logs for the bucket: what objects were read, by request ID, to what egress bytes
  5. Notify: incident commander → legal → privacy/DPO → customer comms if data is confirmed exfiltrated
  6. Long-term: move humans off IAM users to Identity Center with short-lived creds; add a dormant access key Config rule; add GuardDuty anomalous user behavior auto-remediation via EventBridge → Lambda

Console Exploration (read-only)

Navigate to these in the AWS Console. Do not create, edit, or save anything.

  1. GuardDuty → Findings. Open any finding and notice the Actor, Action, Resource tabs
  2. Security Hub → Findings. Browse the severity filter and the Insights tab
  3. Detective → open any existing investigation (if available) — note the timeline and related-entities view
  4. CloudTrail → Event history. Filter Event source = iam.amazonaws.com and look at DeleteAccessKey events
  5. Systems Manager → Documents. Search AWS-DisableEC2InstanceSSH, AWSSupport-* — these are built-in runbooks you can call from EventBridge

Discussion

  • Which of these actions are safe to automate, and which need a human in the loop?
  • How do you practice these playbooks before a real incident? Game days, pre-recorded finding replays, chaos-eng style drills
  • If the compromised role had tag-based access, how much would the blast radius shrink?
